# 附录 A：Hyperliquid 交易规则参考

## 一、交易费率

### 1.1 永续合约费率（按 14 天滚动交易量分层，每日 UTC 评估）

| 层级 | 14 天交易量 | Taker 费率       | Maker 费率       |
| ---- | ----------- | ---------------- | ---------------- |
| 0    | ≤ $5M       | 0.035% (3.5 bps) | 0.010% (1.0 bps) |
| 1    | > $5M       | 0.030% (3.0 bps) | 0.005% (0.5 bps) |
| 2    | > $25M      | 0.025% (2.5 bps) | 0.000%           |
| 3    | > $100M     | 0.023% (2.3 bps) | 0.000%           |
| 4    | > $500M     | 0.021% (2.1 bps) | 0.000%           |
| 5    | > $2B       | 0.019% (1.9 bps) | 0.000%           |

### 1.2 做市商返佣

| MM 层级 | Maker 成交量占比 | 返佣              |
| ------- | ---------------- | ----------------- |
| 1       | > 0.5%           | -0.001% (0.1 bps) |
| 2       | > 1.5%           | -0.002% (0.2 bps) |
| 3       | > 3.0%           | -0.003% (0.3 bps) |

### 1.3 模拟盘费率策略

模拟盘统一使用 Tier 0 费率（Taker 0.035%，Maker 0.01%），不做分层。原因：

- 模拟盘资金量小，不可能达到高层级
- 简化实现，避免维护虚拟交易量统计

## 二、资金费率（Funding Rate）

### 2.1 结算频率

**每小时结算一次**（整点），不同于多数 CEX 的 8 小时周期。

### 2.2 计算公式

```
Funding Rate = Premium Index + clamp(Interest Rate - Premium Index, -0.05%, +0.05%)
```

- Premium Index：永续合约价格与现货指数价格的偏差
- Interest Rate：固定基准利率（约 0.01%/8h，即 ~0.00125%/h）
- clamp 函数将利率与溢价的差值限制在 ±0.05%

### 2.3 费用计算

```
Funding Payment = 持仓数量 × Oracle Price × Funding Rate
```

- Funding Rate > 0：多头付给空头
- Funding Rate < 0：空头付给多头
- 仅在整点持有仓位时收取/支付

### 2.4 模拟盘 Funding 实现

- 每轮 sync round 检查是否跨过整点
- 从 Hyperliquid API 获取真实 Funding Rate
- 按上述公式计算并从 sim_balance 增减
- 记录 Funding 流水到 sim_fills（type 标记为 funding）

## 三、保证金规则

### 3.1 杠杆上限（按资产）

| 资产类型 | 最大杠杆 | 说明                     |
| -------- | -------- | ------------------------ |
| BTC      | 40x      | 2025.3 事件后从 50x 下调 |
| ETH      | 25x      | 2025.3 事件后从 50x 下调 |
| 主流币   | 20x-40x  | 视流动性而定             |
| 小币种   | 3x-10x   | 低流动性资产             |

### 3.2 初始保证金（Initial Margin）

```
初始保证金率 = 1 / 杠杆倍数
```

示例：20x 杠杆 → 初始保证金率 = 5%

### 3.3 维持保证金（Maintenance Margin）

```
维持保证金率 = 最大杠杆下初始保证金率 / 2
```

示例：

- 最大杠杆 40x → 初始保证金率 2.5% → 维持保证金率 1.25%
- 最大杠杆 20x → 初始保证金率 5% → 维持保证金率 2.5%

### 3.4 动态保证金层级

仓位越大，所需保证金比例越高，实际可用杠杆越低。

### 3.5 全仓 vs 逐仓

- **全仓（Cross Margin）**：所有仓位共享一个保证金池，一个仓位的浮盈可支撑另一个仓位的保证金。强平影响整个账户。
- **逐仓（Isolated Margin）**：每个仓位独立保证金，单个仓位强平不影响其他仓位。

### 3.6 模拟盘保证金策略

- 模拟盘默认使用全仓模式（与大多数用户实际使用一致）
- 按上述规则计算初始保证金和维持保证金
- 开仓时检查可用保证金是否充足，不足则拒绝

---

## 四、强平规则

### 4.1 触发条件

账户权益（含浮动盈亏）低于维持保证金要求时触发强平。

### 4.2 Hyperliquid 三级强平机制

1. **市场强平（部分）**：优先通过 orderbook 以市价平仓，支持部分强平——只平掉恢复保证金健康所需的最小仓位
2. **HLP 兜底**：当账户权益低于维持保证金的 2/3 时，HLP（Hyperliquid Liquidity Provider）接管有毒仓位
3. **自动减仓（ADL）**：最后手段，强制盈利方减仓，按利润+杠杆+仓位大小排序

### 4.3 模拟盘强平实现

模拟盘简化为单级强平：

- 当账户权益 < 维持保证金时，按 Mark Price 强制全部平仓
- 剩余保证金归还 sim_balance
- 记录强平事件到 sim_fills
- 不模拟 HLP 和 ADL 机制（模拟盘无需这种复杂度）

## 五、订单类型

Hyperliquid 支持的订单类型：

模拟盘订单处理原则：

- 市价单默认按当前盘口最佳买一 / 卖一直接整单成交
- 由于系统内单笔订单金额受限，当前版本不模拟多档深度、部分成交和额外滑点

| 订单类型                | 说明                         | 模拟盘是否支持 |
| ----------------------- | ---------------------------- | -------------- |
| Market Order            | 按当前盘口最优价立即成交     | ✅ 必须支持    |
| Limit Order             | 指定价格挂单                 | ✅ 必须支持    |
| Stop Market             | 触发价到达后以市价执行       | ✅ 支持        |
| Stop Limit              | 触发价到达后以限价挂单       | ✅ 支持        |
| Take Profit / Stop Loss | 止盈止损                     | ✅ 支持        |
| Scale Order             | 在价格区间内分散挂多个限价单 | ❌ 暂不支持    |
| TWAP                    | 按时间加权分批执行           | ❌ 暂不支持    |

订单修饰符：

| 修饰符          | 说明                         | 模拟盘是否支持 |
| --------------- | ---------------------------- | -------------- |
| GTC             | 挂单直到成交或取消           | ✅             |
| Post-Only (ALO) | 仅做 Maker，会立即成交则拒绝 | ✅             |
| IOC             | 立即成交，否则取消           | ✅             |
| Reduce-Only     | 只能减仓不能加仓             | ✅             |

当前版本的挂单撮合以价格触达为主，不模拟排队优先级。

---

## 六、仓位限制

- Hyperliquid 未公开明确的单用户仓位上限
- 通过动态保证金层级隐性限制：仓位越大保证金要求越高
- 模拟盘额外限制：
  - 单笔订单大小上限（sim_max_order_size，admin 可配）
  - 单用户总仓位价值不超过 sim_balance × 最大杠杆

# 附录 B：API 代理层设计方案

## 一、问题背景

### 1.1 当前架构

```
Agent（本地/远程运行）
    │
    ├──→ 直接调用 Hyperliquid API 下单/查询
    │    POST https://api.hyperliquid.xyz/exchange  (写操作)
    │    POST https://api.hyperliquid.xyz/info      (读操作)
    │
    └──→ 使用自己的私钥 + Vault 地址签名交易

Server（OpenFi 后台）
    │
    └──→ 只做只读数据聚合（sync round 每 60s 拉取一次）
         POST https://api.hyperliquid.xyz/info
```

Agent 直接与 Hyperliquid 通信，Server 不参与交易执行。要实现模拟盘对用户无感知，必须在 Agent 和 Hyperliquid 之间插入一层代理。

### 1.2 核心挑战

- Agent 部署时拿到的是 Hyperliquid API 地址，模拟盘 Agent 不能真的往链上下单
- 用户（和 Agent）不应该知道自己是模拟盘
- 代理层需要完整实现 Hyperliquid 的读写 API 接口
- 实盘 Agent 应该不受代理层影响（可直连或走代理透传）

## 二、整体方案：Hyperliquid API 代理网关

### 2.1 架构

```
Agent
    │
    └──→ POST https://proxy.clawfi.com/exchange   (写操作)
         POST https://proxy.clawfi.com/info        (读操作)
              │
              ├── [实盘 Agent] ──→ 透传到 https://api.hyperliquid.xyz
              │
              └── [模拟盘 Agent] ──→ 路由到模拟撮合引擎
```

### 2.2 路由判断

代理网关收到请求后，通过请求中的签名公钥（或 Vault 地址）查询该 Agent 所属用户的 trading_mode：

- `trading_mode = "live"` → 原样转发到 Hyperliquid API，返回真实响应
- `trading_mode = "sim"` → 交给模拟撮合引擎处理，返回模拟响应（格式与 Hyperliquid 完全一致）

### 2.3 部署方式

代理网关作为 OpenFi Server 的一个新模块，与主服务共享数据库和配置。

两种部署选项：

| 方案        | 说明                                                | 优劣                                  |
| ----------- | --------------------------------------------------- | ------------------------------------- |
| A. 内嵌模块 | 代理网关作为 Server 的一组新路由（如 /hl-proxy/\*） | 简单，共享进程和 DB，但增加主服务负载 |
| B. 独立服务 | 代理网关独立部署，通过 DB 或内部 API 查询用户模式   | 解耦，可独立扩展，但增加运维复杂度    |

建议初期用方案 A（内嵌），后续流量大了再拆分。

## 三、需要代理的 Hyperliquid API 接口

### 3.1 写操作（/exchange）

Agent 下单、撤单、修改订单都走 /exchange 端点。模拟盘需要拦截并模拟的操作：

| 操作           | action type          | 模拟盘处理                         |
| -------------- | -------------------- | ---------------------------------- |
| 下单           | order                | 模拟撮合引擎处理，返回模拟成交结果 |
| 撤单           | cancel               | 从模拟挂单队列中移除               |
| 批量下单       | batchModify          | 逐个处理                           |
| 修改杠杆       | updateLeverage       | 更新模拟账户杠杆设置               |
| 修改保证金模式 | updateIsolatedMargin | 更新模拟账户保证金模式             |

### 3.2 读操作（/info）

Agent 查询账户状态、持仓、成交记录都走 /info 端点。模拟盘需要返回模拟数据的查询：

| 查询类型 | type 参数              | 模拟盘处理                              |
| -------- | ---------------------- | --------------------------------------- |
| 账户状态 | clearinghouseState     | 从 sim_positions + sim_balance 构造响应 |
| 持仓信息 | clearinghouseState     | 从 sim_positions 构造                   |
| 成交记录 | userFills              | 从 sim_fills 查询                       |
| 挂单列表 | openOrders             | 从模拟挂单队列查询                      |
| 现货余额 | spotClearinghouseState | 返回模拟余额                            |

以下查询不需要模拟，直接透传到 Hyperliquid（因为是公共市场数据）：

| 查询类型     | type 参数      | 处理方式                     |
| ------------ | -------------- | ---------------------------- |
| Orderbook    | l2Book         | 透传（模拟盘也需要真实行情） |
| 所有中间价   | allMids        | 透传                         |
| 元数据       | meta           | 透传                         |
| K 线数据     | candleSnapshot | 透传                         |
| Funding 历史 | fundingHistory | 透传                         |

## 四、代理网关响应格式

### 4.1 核心原则

模拟盘返回的响应必须与 Hyperliquid 真实 API 响应格式完全一致，Agent 无法区分。

### 4.2 下单响应示例

Hyperliquid 真实下单响应：

```json
{
  "status": "ok",
  "response": {
    "type": "order",
    "data": {
      "statuses": [
        {
          "filled": {
            "totalSz": "0.1",
            "avgPx": "3500.5",
            "oid": 123456789
          }
        }
      ]
    }
  }
}
```

模拟盘需要返回完全相同格式，其中：

- `totalSz`、`avgPx` 由模拟撮合引擎计算；市价单默认取当时盘口最优价
- `oid` 由模拟系统生成（自增 ID 或 UUID）

### 4.3 账户状态响应示例

需要从 sim_positions 和 sim_balance 构造与 Hyperliquid clearinghouseState 完全一致的 JSON 结构，包括：

- `marginSummary`（accountValue, totalNtlPos, totalRawUsd, etc.）
- `assetPositions`（每个持仓的 coin, szi, entryPx, positionValue, unrealizedPnl, leverage, liquidationPx, etc.）
- `crossMarginSummary` 或 `crossMaintenanceMarginUsed`

## 五、Agent 部署流程变更

### 5.1 当前流程

1. 用户请求部署命令
2. Server 返回包含私钥 + 公钥 + Vault 地址的命令
3. 命令中 Hyperliquid API 地址硬编码为 `https://api.hyperliquid.xyz`

### 5.2 新流程

1. 用户请求部署命令
2. Server 根据用户 trading_mode 决定 API 地址：
   - `trading_mode = "live"` → 可以用代理地址或直连 Hyperliquid（建议统一走代理，方便后续切换）
   - `trading_mode = "sim"` → 必须用代理地址
3. 部署命令中的 API 地址替换为代理网关地址

dispatch_command 模板新增占位符：

- `#hlapi#` → 替换为代理网关地址（如 `https://proxy.clawfi.com`）

### 5.3 模式切换时的 Agent 重部署

当用户从模拟盘切换到实盘（或反向）时：

- Agent 需要重新部署（因为 API 地址可能变化）
- 如果统一走代理网关，则 Agent 不需要重新部署，代理层自动路由
- **建议：所有 Agent 统一走代理网关**，这样模式切换对 Agent 完全透明

## 六、模拟挂单队列

### 6.1 需求

Agent 可能下限价单，这些单不会立即成交，需要维护一个挂单队列。

当前版本的挂单队列只维护未成交订单，成交后直接落 sim_fills，不做部分成交状态跟踪。

### 6.2 新增表：sim_open_orders

| 字段             | 类型    | 说明                                       |
| ---------------- | ------- | ------------------------------------------ |
| id               | TEXT PK | 订单 ID（模拟生成）                        |
| user_id          | TEXT    | 所属用户                                   |
| public_key       | TEXT    | Agent 公钥                                 |
| coin             | TEXT    | 交易对                                     |
| side             | TEXT    | buy / sell                                 |
| size             | REAL    | 订单数量                                   |
| price            | REAL    | 限价                                       |
| order_type       | TEXT    | limit / stop_market / stop_limit / tp / sl |
| trigger_price    | REAL    | 触发价（stop/tp/sl 类型）                  |
| reduce_only      | INTEGER | 是否 reduce-only                           |
| post_only        | INTEGER | 是否 post-only                             |
| time_in_force    | TEXT    | GTC / IOC                                  |
| status           | TEXT    | open / filled / cancelled / expired        |
| created_at       | TEXT    | 创建时间                                   |
| updated_at       | TEXT    | 更新时间                                   |

### 6.3 挂单撮合时机

在每轮 sync round 的 Phase 4（模拟盘数据同步）中：

1. 获取最新 Mark Price 和盘口最佳买一 / 卖一
2. 遍历所有 open 状态的模拟挂单
3. 判断是否满足成交条件：
   - Limit Buy：卖一价格 ≤ 限价 → 按限价成交
   - Limit Sell：买一价格 ≥ 限价 → 按限价成交
   - Stop Market Buy：Mark Price ≥ 触发价 → 按当时卖一成交
   - Stop Market Sell：Mark Price ≤ 触发价 → 按当时买一成交
   - Stop Limit / TP / SL：按同样的“触发后挂限价 / 按最优价成交”原则处理
4. 当前版本默认整单成交，不模拟部分成交、排队优先级和盘口深度
5. 成交后更新 sim_positions、sim_fills，并从 sim_open_orders 移除

## 七、签名验证

### 7.1 问题

Hyperliquid /exchange 接口要求 EIP-712 签名。代理网关需要处理签名。

### 7.2 方案

- **实盘**：代理网关原样转发（包含签名），Hyperliquid 验证签名
- **模拟盘**：代理网关验证签名合法性（确认是该 Agent 的私钥签的），但不转发到 Hyperliquid。验证通过后交给模拟引擎处理

模拟盘签名验证可以简化：只验证签名者公钥与请求中的 vault 地址匹配即可，不需要完整的 EIP-712 验证（因为不上链）。

## 八、实现优先级

### 8.1 MVP（最小可行版本）

1. 代理网关基础框架（路由判断 + 实盘透传）
2. /exchange 下单拦截 → 模拟市价单按盘口最优价成交
3. /info clearinghouseState 模拟响应
4. /info userFills 模拟响应
5. 部署命令模板更新（统一走代理）

### 8.2 第二阶段

6. 限价单挂单队列 + 价格触达成交
7. Stop / TP / SL 订单支持
8. 撤单、改单支持
9. Funding 费用结算
10. 强平逻辑

### 8.3 第三阶段

11. 完整的 openOrders 查询
12. Post-Only / IOC / Reduce-Only 修饰符
13. 杠杆和保证金模式修改
14. 响应延迟模拟（可选）
