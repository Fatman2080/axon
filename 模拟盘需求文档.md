# Clawfi 模拟盘系统需求文档

## 一、概述

### 1.1 背景

项目增加和实盘并行的模拟盘。

### 1.2 核心理念

- 用户进入系统默认为模拟盘，**用户侧无感知**（不区分实盘/模拟盘）
- 后台 admin 可手动将用户切换为实盘或模拟盘
- 模拟盘产生的收益平台照常兑付（目前平台不涉及到用户侧的收益对付和体现，这部分也不涉及功能新增和变更）
- 模拟盘使用真实 Hyperliquid 行情数据，最大程度保真，开仓损失、手续费这些都要参与计算

## 三、模拟撮合引擎

### 3.1 数据源

使用 Hyperliquid 真实市场数据：

- L2 Orderbook 快照（/info 接口，只取最佳买一/卖一）
- 实时成交流（allMids / WebSocket trades）
- 实时 Funding Rate
- 实时 Mark Price

### 3.2 市价单撮合

1. 获取当前 Hyperliquid 盘口最佳买一 / 卖一
2. 买市价单：按卖一价格立即整单成交
3. 卖市价单：按买一价格立即整单成交
4. 当前版本不模拟多档吃单、部分成交和额外滑点

### 3.3 限价单撮合

1. 持续监控真实市场最佳买一 / 卖一和 Mark Price
2. Limit Buy：卖一价格 ≤ 限价时触发成交
3. Limit Sell：买一价格 ≥ 限价时触发成交
4. 触发后默认整单成交，成交价按限价记录
5. 不额外校验该价位真实成交量，也不模拟排队优先级

### 3.4 成交简化约定

- 当前版本不单独模拟滑点和盘口深度
- 由于系统内单笔订单金额有限，默认认为顶档流动性足以覆盖成交
- 若后续出现大额订单场景，再扩展多档撮合和滑点模型

### 3.5 Funding 费用

- 按 Hyperliquid 真实 Funding Rate 每小时结算
- 根据持仓大小和 Funding Rate 从账户余额扣除/增加
- 记录 Funding 费用流水

### 3.6 强平逻辑

- 按 Hyperliquid 保证金规则计算维持保证金
- 使用 Mark Price 判断是否触发强平
- 维持保证金不足时按 Mark Price 强制平仓
- 记录强平事件

### 3.7 保真度分析

| 因素       | 实盘          | 模拟盘         | 偏差控制                        |
| ---------- | ------------- | -------------- | ------------------------------- |
| 价格冲击   | 订单影响市场  | 不模拟         | 虚拟本金小（~100u），影响可忽略 |
| 成交确定性 | 可能部分成交  | 默认整单成交   | 限制单笔订单大小                |
| 延迟       | 网络+撮合延迟 | 几乎为零       | 可加人工延迟                    |
| Funding    | 真实扣费      | 按真实费率模拟 | 几乎无差                        |

## 四、账户资金管理

### 4.1 初始资金

- 新用户注册：模拟盘初始资金跟随现有子账户分配逻辑（约 100 USDC）
- 从实盘切到模拟盘：`sim_balance = 当前实盘账户价值`
- 从模拟盘切到实盘：`实盘初始资金 = 当前模拟盘账户价值`

### 4.2 余额变动

- 开仓：扣除保证金
- 平仓：结算已实现盈亏
- Funding 结算：按真实费率增减
- 强平：结算剩余保证金

### 4.3 平台风控

| 配置项                   | 说明                                         | 建议默认值    |
| ------------------------ | -------------------------------------------- | ------------- |
| sim_total_exposure_limit | 模拟盘总敞口上限（平台最多愿意兜底的总金额） | 需 admin 设定 |
| sim_initial_balance      | 新用户默认模拟盘初始资金                     | 100 USDC      |
| sim_max_order_size       | 单笔最大订单量（保障顶档成交假设成立）       | 需 admin 设定 |

### 4.4 总敞口控制

- 平台维护所有模拟盘用户的总虚拟资金量
- 当总敞口接近上限时，新用户仍可注册但暂不分配模拟盘资金
- admin 可实时查看总敞口使用情况

## 五、数据查询逻辑

### 5.1 用户个人数据

根据用户 `trading_mode` 查询对应数据表：

- `trading_mode = "sim"` → 查询 sim_snapshots、sim_fills、sim_positions
- `trading_mode = "live"` → 查询 agent_snapshots、agent_fills、agent_vaults

用户看到的始终是自己当前模式对应的数据，无需知道是哪种模式。

### 5.2 排行榜（混合）

排行榜将实盘和模拟盘数据合并展示，不标注来源：

- 从 agent_fills + sim_fills 联合计算各 Agent 的 PnL、ROI、胜率、交易次数
- 按选定指标统一排序
- 用户无法区分排行榜中谁是实盘、谁是模拟盘

### 5.3 全局平台指标（仅实盘）

以下指标只统计实盘数据：

- TVL（总锁仓价值）
- 平台总 PnL
- 活跃 Agent 数量（仅实盘）
- 总交易次数（仅实盘）

platform_snapshots 和 treasury_snapshots 只记录实盘数据，不包含模拟资金。

### 5.4 Agent Market（混合）

Agent 市场页面展示所有 Agent（实盘 + 模拟盘），不区分来源。

## 六、模式切换

### 6.1 切换流程

#### 模拟盘 → 实盘

1. Admin 在后台发起切换请求
2. 系统设置 `mode_switch_pending = 1`，`mode_switch_target = "live"`
3. 系统在每轮 sync round 中检查该用户的模拟盘持仓
4. 当所有模拟盘持仓已平仓（空仓状态）：
   - 记录当前模拟盘账户价值
   - 分配实盘 Agent 账户
   - 设置实盘初始资金 = 模拟盘当前账户价值（非固定 100u）
   - 更新 `trading_mode = "live"`
   - 清除切换标记
   - 模拟盘历史数据保留（归档）

#### 实盘 → 模拟盘

1. Admin 在后台发起切换请求
2. 系统设置 `mode_switch_pending = 1`，`mode_switch_target = "sim"`
3. 系统在每轮 sync round 中检查该用户的实盘持仓
4. 当所有实盘持仓已平仓（空仓状态）：
   - 记录当前实盘账户价值
   - 设置 `sim_balance = 当前实盘账户价值`（非固定 100u）
   - 设置 `sim_initial_capital = 当前实盘账户价值`
   - 更新 `trading_mode = "sim"`
   - 清除切换标记
   - 释放实盘 Agent 账户

### 6.2 切换约束

- 同一时间只能有一个待执行的切换请求
- Admin 可随时取消未执行的切换请求
- 切换期间用户正常使用当前模式，不受影响
- 如果用户在切换请求后又开了新仓位，切换不会取消，继续等待空仓

### 6.3 切换通知

- 切换完成后可通过系统记录，admin可在后台看到日志，用户端无感知
- Admin 后台可查看所有待切换用户列表和等待时长

## 七、Sync Round 扩展

### 7.1 现有流程（不变）

- Phase 1：EVM 合约发现（Vault 扫描）
- Phase 2：Hyperliquid 数据同步（实盘账户）
- Phase 2.5：Treasury 快照
- Phase 2.6：Platform 快照（仅实盘数据）
- Phase 3：一致性检查

### 7.2 新增 Phase 4：模拟盘数据同步

对每个模拟盘用户（`trading_mode = "sim"` 且有活跃持仓）：

1. 获取 Hyperliquid 当前市场数据（Mark Price、最佳买一 / 卖一）
2. 更新所有模拟持仓的 Mark Price
3. 重新计算浮动盈亏
4. 检查 Funding 结算时间，到期则按真实 Funding Rate 结算
5. 检查强平条件
6. 保存 sim_snapshots（账户价值快照）
7. 更新 sim_positions

### 7.3 新增 Phase 5：模式切换检查

对每个 `mode_switch_pending = 1` 的用户：

1. 检查当前模式下是否空仓
2. 如果空仓，执行切换逻辑（见第六章）
3. 记录切换结果

## 十二、风险与注意事项

### 12.1 数据隔离

- 模拟盘和实盘数据严格分表存储
- 所有查询必须根据 trading_mode 路由到正确的数据表
- 防止模拟数据污染实盘指标

### 12.2 总敞口控制

- 平台需设定模拟盘总敞口上限
- 实时监控总敞口使用率
- 接近上限时限制新模拟盘用户分配

### 12.3 模拟精度

- 虚拟本金保持较小（~100 USDC），确保模拟保真度
- 限制单笔订单大小，避免顶档成交假设失真
- 当前默认按盘口顶档整单成交，不模拟盘口深度和复杂滑点
- 定期对比模拟盘与实盘同策略的表现差异

### 12.4 模式切换安全

- 切换只在空仓时执行，避免持仓状态下的数据不一致
- 切换时账户价值直接继承，不重置为固定金额
- 切换过程中用户正常使用，不中断服务
